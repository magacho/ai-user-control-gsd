# Phase 02.3: Integração GitHub Copilot e correção de datas nos seats - Context

**Gathered:** 2026-02-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Two deliverables: (1) Integrate GitHub Copilot as an AI tool following the Cursor pattern, using GWS `git_name` custom property for user matching; (2) Fix and complete date fields in the invalid seats listing (first access + last activity).

</domain>

<decisions>
## Implementation Decisions

### GitHub Copilot Sync
- Follow the exact same sync flow as Cursor (fetch seats via API, link each seat to a user)
- GitHubCopilotApiClient already exists from Phase 02.1 — reuse it
- Admin registers GitHub Copilot manually as AiTool (no auto-migration)
- Seats without user match stay with user_id=NULL (appear in invalid seats listing)
- Capture `last_activity_at` from Copilot API during sync (store in AiToolAccount)

### git_name Mapping Flow
- Full flow: Copilot API returns seat with `login` → query GWS for user with custom property `git_name` matching the login → get email from GWS → use email to link/create user in database
- GWS is the intermediary between GitHub login and corporate email (user may not exist in DB yet)
- Matching is case-insensitive (GitHub usernames are case-insensitive)
- Fallback: if git_name match fails, try matching by email as second criterion

### Seat Dates
- Two date fields needed: "Primeiro Acesso" (seat creation date in the tool) and "Último Uso" (last activity/login in the tool)
- Both dates come from the tool APIs, NOT from the database created_at
- Store in AiToolAccount as new fields (createdAtSource, lastActivityAt) — populated during sync
- If API doesn't provide a date, display "-" or "N/A" (no fallback to DB dates)
- Display format: DD/MM/YYYY (Brazilian standard, consistent with rest of application)

### Listing UI
- New date columns appear in BOTH sections (seats to remove + external/invalid seats)
- Visual highlight for seats inactive > 60 days (last activity older than 60 days)
- Both new date columns are sortable (click header to sort)

### Claude's Discretion
- Exact visual style for inactivity highlight (color, icon, badge)
- Column ordering in tables
- Flyway migration version numbering
- Internal field naming conventions for the new date columns

</decisions>

<specifics>
## Specific Ideas

- The GWS custom property `git_name` is the key link between GitHub identity and corporate identity — this is the primary lookup path, not email
- The user may not exist in the database when a Copilot seat is found — the flow must handle user creation from GWS data after git_name resolution
- Cursor integration (Phase 02.2) is the reference implementation — follow its patterns closely

</specifics>
